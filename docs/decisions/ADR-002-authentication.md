# ADR-002 — Authentication Strategy

**Date:** 2026-05-30
**Status:** Accepted
**Deciders:** Rowlex (Founder/Lead)

---

## Context

Perxli has two authentication surfaces:

1. **Creative-side** — full authenticated app where creatives manage their projects, documents, and payments.
2. **Client-side** — a public link where the creative's client reads documents and types-to-sign. Clients do **not** create accounts. They must not be forced through signup just to sign a document — that friction would kill the product.

The creative-side flow needs to be secure and revocable for compliance (signed documents are legally meaningful). The client-side flow needs to be approachable but still tamper-resistant.

---

## Decision

### Creative-side: JWT Access Tokens + Rotating Refresh Tokens in httpOnly cookies

#### Access Token
- Short-lived: **15 minutes**
- Signed with `RS256` (asymmetric) — allows future microservice verification without sharing the secret
- Payload: `{ sub: userId, role, plan, iat, exp }` (plan included for fast feature-gating without a DB lookup)
- Returned in the response body; stored in memory on the frontend (not localStorage)

#### Refresh Token
- Long-lived: **7 days**
- Stored in an **httpOnly, Secure, SameSite=Strict cookie** — inaccessible to JavaScript
- Stored as a hashed value in the `user_sessions` table (one per device/session)
- **Token rotation:** each use issues a new refresh token and invalidates the old one
- Reuse of an old refresh token triggers full session invalidation (stolen token detection)

#### Session Storage
```sql
user_sessions (
  id, user_id, token_hash, expires_at,
  created_at, last_used_at, ip_address, user_agent
)
```

#### Logout
- Delete the session record from DB
- Clear the cookie
- Frontend discards the in-memory access token

---

### Client-side: Opaque Share-Link Tokens (no auth)

The client receives a URL like:

```
https://perxli.com/s/{token}
```

Where `{token}` is a 32-byte URL-safe base64 random value generated by the server. It is **not** a JWT.

#### Properties
- Stored in `project_share_links (project_id, token, expires_at, revoked_at, view_count, last_viewed_at)`
- Backend looks the token up on every request — no JWT verification, no claims
- One-way: the token grants read access to the project's documents and the ability to type-to-sign. It does **not** grant access to anything else (no creative profile, no other projects, no settings).
- Token can be revoked by the creative at any time; revocation is immediate
- Rate-limited per IP and per token to slow enumeration / brute force

#### Why not JWT here?
Because the client never logs in, there is no refresh ceremony to amortize JWT verification. A DB lookup on each request is acceptable at expected traffic, and gives us instant revocation and view tracking for free.

---

## Alternatives Considered

| Alternative | Reason Rejected |
|-------------|----------------|
| Session cookies only (creative side) | Harder to scale horizontally; requires sticky sessions |
| Access token in localStorage | Vulnerable to XSS attacks — unacceptable when payment + signature actions are protected by it |
| Long-lived JWTs (no refresh) | Cannot revoke tokens; unacceptable when signed-document audit trails depend on session integrity |
| OAuth2 / third-party SSO | Valid future addition (Google for creatives), but adds complexity before we have users |
| Require clients to register/sign up to view documents | Would kill conversion. The whole product proposition is "client just opens the link and signs" |
| Email magic-link for clients | Adds an unnecessary roundtrip; the share link itself is already the secret. Could be added later as a defense-in-depth layer for high-value contracts. |

---

## Security Considerations

### Creative side
- Passwords hashed with `bcrypt` cost factor 12 minimum.
- Rate limit login endpoint: 5 attempts per 15 minutes per IP.
- Log failed login attempts per user (alert on brute force).
- HTTPS enforced in all environments; cookies require `Secure` flag in production.
- Include `jti` (JWT ID) claim for future revocation list if needed.

### Client side
- Share-link tokens are 32 bytes from `crypto.randomBytes` — not guessable.
- Rate limit: 60 requests per IP per minute on `GET /api/v1/share/:token`.
- Tokens default to expiring 60 days after creation; creative can extend or revoke.
- Every signature record stores `ip_address`, `user_agent`, and the signed document's content hash — provides a tamper-evident audit trail even though the signer is unauthenticated.
- AI-generated document HTML is sanitized server-side before being returned to the client view (XSS defense — a malicious project description could otherwise inject script tags).

---

## Consequences

- Frontend stores access token in memory (Zustand). Lost on page refresh — Axios interceptor silently refreshes via the cookie.
- Backend must validate token signature and expiry on every protected request.
- Logout must reach the server to invalidate the session; client-side-only logout is not sufficient.
- The client-side surface has its own controller/middleware path (`shareTokenAuth` instead of `requireUser`) — keep them separated.
- Future SSO (Google Workspace, Microsoft Entra) can be layered on as an additional grant type without replacing this flow.
